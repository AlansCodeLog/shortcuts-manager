addHook

  • addHook<TType, THook>(type: TType, hook: THook): THook

  • Add a hook.

    This works just like adding an event hook:

    const allowsHook = ... // keep a reference if you want to remove it later
    shortcut.addHook("allows", allowsHook)

    if (shortcut.allows(...).isOk) // your hook will fire

    Note that typescript can only understand the types of the errors of the builtin hooks. Therefore if you're adding hooks afterwards, you will need to pass the possible errors the hook can return as the first type parameter if you want them typed correctly:

    class MyError extends Error {
         property: boolean
    }
    let allowed = shortcut.allows<MyError>(...)

    Note some classes might allow adding only some types of hooks, but not others on some properties. For example, you can add set hooks to some properties of collection classes but not allows hooks to those properties.

allows

  • allows<TKey>(key: TKey, value: ShortcutHooks[TKey]["excludeAllows"] extends true ? never : ShortcutHooks[TKey]["value"]): Result<true, Error | ShortcutHooks[TKey]["error"]>

  • Tells you whether a property is allowed to be set.

    Returns a result monad. See Result from my utils lib.

    const res = shortcut.allows("keys", [[key.a]])
    if (res.isOk) {
        shortcut.set("keys", [[key.a]])
    } else { // res.isError
        console.log(res.error.message)
    }

    For the manager, it has a special "property" "replace" for replacing one or more of some of it's properties (keys, commands, or shortcuts) at a time. This is because if you want to replace two of them at a time you could get errors that you would not get if you had tried to do it seperately.

    // examples:
    const res = manager.allows("replace", {shorcuts})
    // same as:
    const res = manager.allows("shortcuts", shorcuts)

    // multiple properties:
    const res = manager.allows("replace", {keys, commands})
    const res = manager.allows("replace", {keys, commands, shortcuts})

conflictsWith

  • conflictsWith(shortcut: Shortcut, __namedParameters?: {
        context?: Context<RecursiveRecord>;
        ignoreChainConflicts?: boolean;
        ignoreModifierConflicts?: boolean;
        ignoreSelf?: boolean;
    }): boolean

  • A shortcut conflicts with another if their conditions are equal (according to this shortcut's condition) and their chains are in conflict.

    Chains can be in conflict in any of the following situations:

    • The shortcuts are the same instance (unless ignoreSelf is passed).
    • All the keys are the same.
    • They share starting chords and the last shared chord conflicts. (e.g. [[A], [B]] and A), i.e. they have a chain conflict.
    • The last chords shares all modifiers and one of the chords is all modifiers. (e.g. [[A], [Ctrl]] and [[A], [Ctrl, A]]), i.e. they have a modifier conflict.

    Some of these can be ignored with the experimental ignore* options.

    A context can be passed for a more accurate test, otherwise depending on what type of conditions you use (see ["equals"]) you cannot truly tell if shortcuts with complex conditions will conflict. Even if a context is passed, both shortcuts should match / eval to true in the context. See the option itself for more details.

    Parameters

    • shortcut: Shortcut
    • __namedParameters: {
          context?: Context<RecursiveRecord>;
          ignoreChainConflicts?: boolean;
          ignoreModifierConflicts?: boolean;
          ignoreSelf?: boolean;
      } = {}
      • Optional Experimental context?: Context<RecursiveRecord>

        If a context is passed, given all other conditions are equal, the shortcuts can only be in conflict if they both match the context. Note this means if neither match the context they are not considered to conflict (even though they might in another context). This is meant to be used when they both match the context, but some behavior had to be chosen for when they don't.

      • Optional Experimental ignoreChainConflicts?: boolean

        If this is true, partial chain conflicts (e.g. Space conflicts with Space+A) will be ignored. You will need to figure out how to handle them manually.

      • Optional Experimental ignoreModifierConflicts?: boolean

        If this is true, modifier conflicts (e.g. Ctrl conflicts with Ctrl+A) will be ignored. You will need to figure out how to handle them manually.
